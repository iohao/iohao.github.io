Property Change Listener

Introduction​

Properties can attach listeners. When a property value changes, listeners are triggered.


Property listener features

  • Add listeners to properties to observe value changes.
  • A property can have multiple listeners.
  • Property listeners can be removed.

The framework has built-in property implementations:

  • IntegerProperty
  • LongProperty
  • StringProperty
  • BooleanProperty
  • ObjectProperty

In addition to built-in types, developers can extend more types themselves.

Usage Scenarios​

Scenario 1

For example, when a player's HP drops below a threshold, invincible state should be triggered. In this case, you can listen to player HP and add a listener on that property to observe HP changes. When the expected value is reached, trigger corresponding business logic.


Scenario 2

This feature can also manage room states. When room state changes, listeners execute corresponding business logic.


There are many similar scenarios. The key point is that listener callbacks are triggered after property changes.

How to Use​

BooleanProperty​

When the value of a BooleanProperty changes, the listener is triggered.

  • code 2: add a listener.
  • code 7: when value changes, listener is triggered.
var property = new BooleanProperty();
property.addListener((observable, oldValue, newValue) -> {
   log.info("oldValue:{}, newValue:{}", oldValue, newValue);
});

property.get(); // value is false
property.set(true);
property.get(); // value is true

IntegerProperty​

When the value of an IntegerProperty changes, the listener is triggered.

  • code 2: add a listener.
  • code 7: when value changes, listener is triggered.
  • code 10: convenient method to increase IntegerProperty value.
var property = new IntegerProperty();
property.addListener((observable, oldValue, newValue) -> {
   log.info("oldValue:{}, newValue:{}", oldValue, newValue);
});

property.get(); // value is 0
property.set(22);
property.get(); // value is 22

property.increment(); // value is 23. will trigger listeners

ObjectProperty​

When the reference of the internal value in an ObjectProperty changes, the listener is triggered.

  • code 4: add a listener.
  • code 8: listener is not triggered, because the reference does not change.
  • code 11: reference changes, triggering listener.
YourUser user = new YourUser();

var property = new ObjectProperty(user);
property.addListener((observable, oldValue, newValue) -> {
   log.info("oldValue:{}, newValue:{}", oldValue, newValue);
});

property.set(user); // does not trigger listeners

YourUser user2 = new YourUser();
property.set(user2);
property.get(); // value is user2

class YourUser {}

Remove Listener​

In the following sample, initial property value is set to 10, then a listener is added. When the listener observes new value 9, it removes itself from observable (observable here is IntegerProperty).

  • code 4: add a listener.
  • code 10: remove current listener.
  • code 15: value minus 1, current value is 9, listener is triggered.
  • code 16: value minus 1, current value is 8, but listener is not triggered because it has been removed.
public void remove1() {
    IntegerProperty property = new IntegerProperty(10);

    property.addListener(new PropertyChangeListener<>() {
        @Override
        public void changed(PropertyValueObservable<? extends Number> observable, Number oldValue, Number newValue) {
            log.info("1 - newValue : {}", newValue);

            if (newValue.intValue() == 9) {
                observable.removeListener(this);
            }
        }
    });

    property.decrement(); // the value is 9
    property.decrement(); // the value is 8. since the listener is removed, no event is triggered.
}

In the following example, we define listener class OnePropertyChangeListener implementing PropertyChangeListener. We remove the listener through the reference of OnePropertyChangeListener object.

  • code 2: create custom listener.
  • code 5: add listener.
  • code 7: value plus 1, current value is 1, and listener is triggered.
  • code 8: remove listener.
  • code 9: value plus 1, current value is 2, but listener is not triggered because it was removed.
  • code 12: custom listener.
public void remove2() {
    OnePropertyChangeListener onePropertyChangeListener = new OnePropertyChangeListener();

    IntegerProperty property = new IntegerProperty();
    property.addListener(onePropertyChangeListener);

    property.increment(); // the value is 1
    property.removeListener(onePropertyChangeListener);
    property.increment(); // the value is 2
}

class OnePropertyChangeListener implements PropertyChangeListener<Number> {
    @Override
    public void changed(PropertyValueObservable<? extends Number> observable, Number oldValue, Number newValue) {
        log.info("oldValue:{}, newValue:{}, observable:{}", oldValue, newValue, observable);
    }
}

Summary​

Property listening is simple to use. If your business requires triggering events after property changes, this feature is worth considering. The framework provides corresponding property listeners for base types such as int, long, boolean, Object, and String.

Property listening supports adding multiple listeners and removing listeners.

Further Reading​

Case: Monitor Room Game State​

Listen to different room game states and trigger business methods.

public class FairRoom extends SimpleRoom {
    ...
    final ObjectProperty<GameStatus> gameStatus = new ObjectProperty<>(GameStatus.NONE);

    private void extractedInitGameStatus() {
        this.gameStatus.addListener((observable, oldValue, newValue) -> {
            switch (newValue) {
                case START -> nextRound();
                case SETTLE -> settle();
            }
        });
    }

    public void changeGameStatus(GameStatus status) {
        this.executeTask(() -> this.gameStatus.set(status));
    }

    public void changeGameStatus(GameStatus status, long millis) {
        this.executeDelayTask(() -> this.gameStatus.set(status), millis);
    }
}

public enum GameStatus {
    NONE,
    START,
    SETTLE
}

Case: Combined with Dynamic Attributes​

This is a case combining property listeners and dynamic attributes.


Business description

In a simulated client, we want to count heartbeat sends and trigger custom logic after a threshold, such as closing the simulated client.

This can be done by combining property listeners and dynamic attributes. Add dynamic property idleCounter to clientUser. idleCounter is an IntegerProperty. Attach a listener so when heartbeat count is greater than 10, disconnect simulated client from server.

When property listeners are combined with dynamic attributes, this special requirement can be implemented easily.

Since simulated client does not provide heartbeat-count property out of the box, add one through dynamic attributes for clientUser. If business requires triggering custom logic after heartbeat reaches a threshold, monitor heartbeat change using xxxProperty (property listener).

static void main() {
    int idlePeriod = 3;
    // simulated player
    var clientUser = getDefaultClientUser(idlePeriod);

    new ClientRunOne()
            .setClientUser(clientUser)
            // enable heartbeat, send heartbeat to server every N seconds
            .idle(idlePeriod)
            .startup();
}

private static DefaultClientUser getDefaultClientUser(int idlePeriod) {
    // dynamic attribute key
    AttrOption<IntegerProperty> idleOption = AttrOption.valueOf("idleOption");
    // simulated heartbeat counter
    IntegerProperty idleCounter = new IntegerProperty();
    // add listener, triggered when value changes
    idleCounter.addListener((observable, oldValue, newValue) -> {
        if (newValue.intValue() > 10) {
            clientUser.getClientUserChannel().closeChannel();
        }
    });

    // add dynamic attribute to clientUser
    var clientUser = new DefaultClientUser();
    clientUser.option(idleOption, idleCounter);

    // execute every N seconds
    TaskKit.runInterval(() -> {
        // heartbeat counter +1, trigger listener on each value change
        clientUser.ifPresent(idleOption, IntegerProperty::increment);
    }, idlePeriod, TimeUnit.SECONDS);

    return clientUser;
}
